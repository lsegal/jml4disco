1. Introduction

This page describes the process upheld by the developers and maintainers of this project. Specifically this document describes protocols followed by those using the Trac project management system and the Subversion versioning management system in addition to rules about code formatting and testing.

All of these guidelines are subject to change should they hinder the team's efficiency or performance. Please contact a Project Manager (PM) if you have an issue with a guideline described below.

2. Trac Protocols

2.1 Updating the Wiki

All team members are given full permission to add or modify any page on the Wiki, as long as they adhere to the following guidelines:

  • Only add content that is relevant to the project scope.
  • Only modify content that is under your responsibility, unless you are correcting or improving the document without modifying its original intent.
  • Add a descriptive comment (located in the change information section of a modify page form) for every change made.
  • All changes related to open tickets must be reviewed by the ticket reviewer before closing the ticket in question.

In addition to these guidelines, communication should be made between members to ensure that no two people are working on the same page at a given time. While documents can be merged after concurrent changes are made, this is a generally complicated process that should be avoided with proper communication.

2.2 Managing tickets and milestones

Tickets that contribute to a milestone must be opened by the respective activity category leader. Leaders are indicated on the Resource / Activity table in the Project Plan. Team members have permission to add information to any ticket. Tickets may be re-opened by the reporter, the assignee, or the reviewer. Tickets must be reviewed and closed by the reviewer.

Tickets that do not contribute to a milestone, as well as milestones themselves, are to be created by the PM or AM.

A team member can contact a manager or activity leader to request that a ticket be re-assigned to another member.

2.3 Ticket Closing and Review Process

When a user is finished doing work on a ticket and is ready to have it reviewed, he should mark the ticket as "Ready for review" (in the ticket's properties) and wait for the reviewer to review the ticket and work done.

If a reviewer is satisfied with the work, he will be responsible for closing the ticket, marking it as fixed (or whichever action is most relevant to the ticket) and setting the "Completion date" field. If a reviewer believes the work is not satisfactory, he should unset the "Ready for review" status (in ticket properties) and then contact the owner to make the necessary adjustments. In this secondary case, the owner will make the changes and the scenario will be restarted from the beginning.

Please note that it is the reviewers responsibility to close the tickets, not the owner's. However, after the ticket has been closed the owner must go back and set their "Effort" in person days in the ticket.

3. Subversion Protocols

3.1 Checking out source code

All members can check out any source code they wish. All code submitted to the repository should be visible and modifiable by all members.

3.2 Checking in source code

All members should please follow the Subversion conventions when submitting code to the repository:

  • All project code specific to the latest build should reside in trunk/
  • All code under experimental stage should be moved to branches/ with a directory name describing the feature the project is implementing. Moving should only be done with the svn copy command.
  • When code is ready for a milestone or release it should be copied with svn copy to tags/ with the sub-directory name representing its final version number (or symbolic release name). Only PM's or AM's should be responsible for this operation.
  • No member should ever commit code to tags/. The only way code should ever be added to the tags/ directory is via the svn copy command.
  • Members should only check in files that are atomically complete. In other words, the member has finished making a change that can exist independently on its own. In terms of source code, this means not committing code unless it passes all unit tests and will not break the current build in the repository. This guideline is less strict for data files, but every committer should remember that Subversion should not be used as temporary disk storage-- but storage for completed items.
  • Every commit should attach a descriptive message specifying the changes made to the codebase.
  • Each message should also attach a ticket id to reference which ticket this code is meant for.
  • Except for maintenance commits (fixing broken directory structures or missing files), all commits should reference a valid ticket id, or they probably do not belong to the trunk/ codebase.
  • A commit message should have the following syntax (Trac will automatically format the message accordingly, please see WikiFormatting on how to link to tickets): 
    Added error messages to the login screen. (See ticket #438)
  • If multiple files are modified referencing multiple tickets, it would be best to perform two separate commits for each set of files and reference them individually, though the following message syntax is also permissable: 
    - Fixed user upload form to allow files over 2MB. (See ticket #95)
- Fixed name validation in user registration to be more lenient about input (See ticket #80)
  • All changes related to open tickets must be reviewed by the ticket reviewer before closing the ticket in question.

In addition to these Subversion conventions, users should be aware of filename conventions used in the project. Filenames and directories should attempt to follow UNIX naming conventions. That is, files and directories should avoid the use of spaces, and, to a lesser extent, non-alphanumeric characters ([^A-Za-z0-9_]). Files and directories should also be in lowercase except for files that must abide by Java's naming conventions, such as .java and .class files.

Please remember not to commit any sensitive data to the repository such as personal passwords or trade secrets, as this data cannot be [easily] removed after a commit. Refer to the svn ignore functionality to ignore files of this type from being committed to the repository. This includes setting an svn ignore on files such as the .project file in the eclipse directory, as other users may have different project settings that may collide with yours. You can ignore the .project file by typing (in a terminal window inside your local trunk directory): 

 svn propset svn:ignore .project .

Do the same to any other files that require specific changes to suit your development environment.

3.3 Merging files and handling collisions

In the case of a merging issue in Subversion, team members should communicate amongst themselves to resolve the collision in question. If the merge is trivial, one member can perform it alone, as long as it does not break the build, but the member should notify the other member of the change.

4. Code and Testing Protocols

4.1 Source code etiquette and formatting rules

Java Code should follow standard Eclipse Java formatting rules (in Eclipse press SHIFT-CTRL-F to format the code). To summarize the rules:

  • Code should be indented with 4 spaces (space indentation, not tabs)
  • Opening braces should reside on the same line as the method or block. Example: 
    public static void main(String[] args) {
    return 0;
}
  • All methods should be commented with Javadoc compatible comments. Please brush up on your Javadoc abilities.
  • One line comments should use //. Multi-line comments should use /* */

Please do not commit uncommented code. Procrastination in commenting will only perpetuate a documentation problem in the project. Keep in mind that you are writing code for others, not yourself. Think of each stakeholder every time you commit your code.

4.2 Managing unit tests

Please follow the following guidelines when dealing with testing during the implementation phase:

  • All written code should come with at least one extra unit test added to the test base.
  • All tests should pass before committing code.
  • Do not commit code unless a test has been written for it.
  • Make sure that the tests are as legible as the application logic it tests. An incorrect test is just as useless as no tests at all.
