1. Introduction

1.1 Purpose

Software development and maintenance are costly endeavors. The cost can be reduced if more software defects are detected early in the development cycle. [3] This can be achieved using a formal behavioral interface specification language for Design by Contract (DBC) as well as compile-time checks to assert that contracts are respected. The Java Modeling Language (JML) is one such language, and tools have been developed to perform static checking of JML-enhanced Java programs. The Dependable Software Research Group (DSRG) has developed the JML4 Integrated Verification Environment (IVE); an Eclipse-based JML tool which performs Extended Static Checking. One challenge facing JML4 is the reality that applying ESC to industrial-scale applications has been - and will be - difficult due to the time it takes to perform static check, as well as the incomplete nature of tools which do so. [1] JML4 Disco will be an evolution of JML4 meant to alleviate these problems by improving ESC4's potential for distributed application verification, to better utilize non-local resources and reduce the time to assert the validity of software components, and to achieve more complete coverage of the Java language then would be possible using ESC4 alone. The purpose of this document is to define the user needs, the system features, and the project requirements for JML4 Disco.

1.2. JML4 Disco Overview

JML4 Disco will be a system built on top of the existing JML4 system with relevant changes made to the system such that JML4 Disco will be able to perform distributed application verification as well as offer more complete Java coverage.

1.2.1 Distributed Application Verification

Applying Extended Static Checking to large-scale application has been difficult because existing tools require significant amounts of time to verify annotated code; while Extended Static Checking is generally quite an efficient technique, it is still orders of magnitude slower than simple compilation.[1] Given that the underlying nature of the verification techniques of extended static checking are inherently modular, distribution of the verification could improved performance considerably. [1] As such the first key feature of JML4 Disco will be improved distributed discharging of application information.

1.2.2 More Complete Java Coverage

The other key feature of JML4 Disco will be to provide more complete coverage of the Java language then is currently possible using the underlying ESC4 back-end. This will be achieved by developing a means of interfacing JML4 Disco with another existing application verifier.

2. Positioning

2.1 Product Statement

The Problem of: There is a lack of efficient and reliable extended static checking tools for Java programs.
Affects: Java developers engaged in quality control via Design By Contract (DBC) using JML and ESC as well as do DC Extended Static Checking (ESC)
The impact of which is: Applying program verification - which could reduce the cost of development and maintenance - to industrial-scale Java applications has been difficult.
A successful solution would be: An evolution of JML4 to include more complete Java coverage while improving ESC4's potential for distributed application verification.

For Java Developers and the ESC4 team
Who Engage in quality control via DBC using JML and ESC, as well as the Eclipse IDE
JML4Disco Is an improved version of the JML4 project with faster, more complete Java-program verification capabilities
That Will allow for practical and cost-effective static checking of industrial-scale Java applications, discovering software errors earlier in the development cycle.
Unlike Discovering software faults or errors at a higher cost later in the development cycle.
Our Product Will verify applications quickly as well as have a more complete coverage of the Java language.

3.2 User Environment

A Java developer wanting to do engage in Design by Contract using JML Disco to verify annotated code will typically do so through the Eclipse Ganymede Environment. Multiple servers can be setup to perform and contribute to remote code verification, as a single client may distribute the modular tasks involved in Extended Static Checking to multiple machines.

4.4 Alternatives and Competition

4.4.1 Contracts for Java

Contracts for Java (C4J) is a Design By Contract Java. The primary goal for C4J is ease of use. TSascha Zelzer is developing an Eclipse plugin for C4J, which is available on SourceForge.

Contract4J provides a standard way to define DbC tests and to evaluate them at runtime. Using XDoclet tags and generating AspectJ test code. Contract4J uses Java 5 annotations to define the condition tests and it uses "prebuilt" AspectJ aspects to advise the execution points ("join points") where the tests should be evaluated. If the tests fail, both tools terminate program execution. [2]

4.4.2 First Generation JML Tools

4.4.4.1 Common JML Tool suite (JML2)

JML2 is the name of the current release of the JML Common Tools. It nearly completely supports Java 1.4.2, and includes limited support for Java 5+, and can perform runtime assertion checking as well as static checking of most of the features of the JML language. [6]

4.4.4.2 ESC/Java2

ESC/Java (and more recently ESC/Java2), the "Extended Static Checker for Java," is a programming tool that attempts to find common run-time errors in Java programs by static analysis of the program text. Recent versions of ESC/Java are based around the Java Modeling Language (JML). Users can control the amount and kinds of checking by annotating their programs with specially formatted comments or pragmas. [4]

ESC/Java was originally developed at the Compaq Systems Research Center (SRC). SRC launched the project in 1997, after work on their original extended static checker, ESC/Modula-3, ended in 1996. In 2002, SRC released the source code for ESC/Java and related tools. [4]

The University of Nijmegen's Security of Systems group released alpha versions of ESC/Java2, an extended version of ESC/Java that processes the JML specification language through 2004. Since 2004, ESC/Java2 development has been managed by the KindSoftware Research Group at University College Dublin. Over the years, ESC/Java2 has gained many new features including the ability to reason with multiple theorem provers and integration with Eclipse. [4]

It apparently not sound and not quite complete, sometimes giving false positives and false negatives, though it finds lots of potential bugs quickly. Additionally, Using ESC/Java2 is only really feasible on small programs. [5]

4.4.4.3 PVS Tool

PVS is an environment for formal methods and specifications. PVS consists of a specification language, predefined theories, a type checker, and an interactive theorem prover, formalized libraries. It is a research prototype, and the most recent release is version 4.2. It is currently open source and can be run under most Linux based systems. 1. 5. Other Product Requirements

5.1 Applicable Standards

JML4 Disco is an evolution of JML4. Given that JML4 is an Eclipse-based Integrated development Verification Environment, built on top of the Eclipse Java Development Tooling, JML4 Disco should be roughly complaisant with the Eclipse Development Conventions and Guidelines.

5.2 System Requirements

For the client, a typical modern desktop or laptop is adequate to run the application. For the servers, it will be better to have a reliable and more powerful resources to handle large amount of requests without diminishing the quality of services.

5.3 Licensing, Security and Installation

The Project is an Open Source Project. There are no security requirements in this project.

5.4 Performance Requirements

JML4 Disco must perform faster than JML4; timed results of verification attempts should be faster than those of the prototype Distributed Multi-Threaded Verifier Prototype developed by the DSRG. [1]
