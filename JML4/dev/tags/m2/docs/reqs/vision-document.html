1. Introduction

The Vision document defines our team's vision for the JML Disco Project. In the project we will aid the DSRG team in developing solutions intended for JAVA developers using the Java Modeling Language (JML) to do design by contract (DBC). We will contribute to the project by working on 2 features to further improve JML4. One is to improve the implementation of distributed discharging of verification conditions (VCs). Extended Static Checking (ESC) is a fully automated formal verification technique and is generally quite efficient, as far as verification tools go, but it is still orders of magnitude slower than simple compilation. Verification in ESC is achieved by translating programs and their specifications into verification conditions (VCs). Proof of a VC establishes the correctness of the program but proving VCs is computationally expensive. (1) To help address this lack of scalability, we will improve the the current version of ESC4 and its distributed prover back-end. The other key feature is to integrate Boogie in the project.Boogie is a verification language, designed to make the prescription of verification conditions natural and convenient. JML Disco will translate JML-decorated AST to Boogie, use it as a verification tool and use its output to report errors/warnings.

2. Positioning

2.1 Product Statement

The Problem of: Lack of efficient and reliable extended static checking of Object Oriented Java through JML4
Affects: Java developers using JML and ESC and to do design by contract (DBC) and do DC ESC
The impact of which is: Time lost generating (sometimes unattainable) results using ESC4 provers
A successful solution would be: Evolve the current prototype system with better distributed strategies and added Boogie compatibility.

User Environment

A Java developer wanting to do do DBC using JML4 with JML4 Disco will typically do so through the Eclipse Ganymede Environment. Multiple servers can be setup to act as provers as a client using Eclipse may distribute the discharging of VCs on to these multiple machines. These servers will have Tomcat Servlets ready to receive VCs and process them. These servers can be run locally or remotely although prototyping and test results have only been generated for distributed discharging of VCs over LAN.

Alternatives and Competition

Contracts for Java (C4J) is a Design By Contract Java. The primary goal for C4J is ease of use. TSascha Zelzer is developing an Eclipse plugin for C4J, which is available on SourceForge?.

Contract4J provides a standard way to define DbC tests and to evaluate them at runtime. Using XDoclet tags and generating AspectJ test code. Contract4J uses Java 5 annotations to define the condition tests and it uses "prebuilt" AspectJ aspects to advise the execution points ("join points") where the tests should be evaluated. If the tests fail, both tools terminate program execution.(2)
