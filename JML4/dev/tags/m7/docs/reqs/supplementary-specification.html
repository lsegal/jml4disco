1. Introduction

1.1 Purpose

The purpose of this document is to collect all non-functional requirements and design constraints placed on the JML4 Disco project, as well as any functional requirements not expressed in the Use Case Model.

1.2 Scope

This document is relevant to the JML4 Disco project. JML4 Disco will be an evolution of the Dependable Software Research Group's JML4 Integrated development and Verification Environment, meant to alleviate the problem of applying ESC to industrial-scale applications, by means of improving ESC4's potential distributed application verification, so as to better utilize non-local resources and reduce the time to assert the validity of software components and interfacing with Microsoft's state-of-the-art program verifier Boogie.

1.3 Definitions, Acronyms, and Abbreviations

Please see the Glossary for all definitions, acronyms, and abbreviations.

2. Functional Requirements

The following are a list of functional requirements that the system provides. The feature summary document describes in details the current status of the design of the significant features as well theirs limitations.

2.1 Multi-threaded Verification

JML4 Disco application will take the advantage power of multi-core architecture to prove the various program unit concurrently to improve the performance of proving process.

2.2 Distributed Discharing VC

One of the needs of the stakeholder is to improve the performance of the program verification process. To satisfy this need, JML4 Disco application provides the ability to verify the program distributively over either LAN or WAN.

2.3 Load Balancing

The distributed program verification capabilities of JML4 Disco should be able to perform dynamic load balancing so as to optimize the use of remote resources.

2.4 Verification Unit Caching

Given that network communication can be costly, the distributed program verification capabilities of JML4 Disco should be able to cache results which have been unchanged. Cached results will not make need for remote resources.

2.5 User-Managed Theorem Provers Process

Control properly and efficiently the number of processes been spawn on the remote prover hosts in order to avoid overhead of creating/deleting processes and context switching.

2.6 Integration of Boogie Prover

To make proving more complete, the JML decorated Java source will need to be translated to Boogie and then checked for correctness. Using Boogie as a means of extending proving was stated as a requirement by the stakeholder. [Details]

3. Usability

A user who is familiar with the semantics of the JML modeling language as well as the Eclipse Integrated Development Environment should require minimal - if any - additional training when using JML4 Disco. In order for JML4 Disco to be consistent with the rest of the Eclipse Integrated Development Environment, any additional User-Interface components developed should conform to the Eclipse Use Interface Guidelines.

4. Reliability

4.1 Availability

The distributed environment of the system should be available at any time to allow client to request services without problem. That is the VC dispatchers and provers hosts must be available 24 hours a day, 7 days a week.

4.2 Recoverability

If a server is down, the system administrator should be aware of it and restore the server as soon as possible. Client's application should not be affected by server crashes and should be able to request for proving VC from other working servers.

5. Performance

JML4 Disco must improve upon the performance of JML4, and JML4 Disco should at least be as responsive as the Prototype Distributed Multi-Threaded Java Program verifier developed by the DSRG. [1] The following are the timing results from that prototype, validating a single Java class with 51 methods: [1]

Number of Servers Number of Cores Time to prove Locally Time to prove Remotely
1 4 26.6 26.4
2 8 16.9 16.2
3 12 12.8 13.3

6. Supportability

6.1 Configurability

JML4 Disco should be able to connect to and make use of resources on varying remote machines based off information in configurations files. JML4 Disco should also make use of varying proving strategies.

6.2 Scalability

The distributed application verification capabilities of a JML4 Disco client should be able to connect to and make use of resources on multiple remote machines.

7. Constraints

7.1 Design Constraints

JML4 Disco must provide a user with the ability to perform program verification making use of resources on remote machines. JML4 Disco must improve performance of Extended Static Checking using caching of previous application verification results. JML4 Disco is constrained to using Microsoft's state-of-the-art Boogie program verifier to achieve more complete coverage of Java.

7.2 Implementation Constraints

Since JML4 Disco is an evolution of JML4, JML4 Disco must be implemented in Java. The JML4 Disco will solely integrate with Eclipse IDE. Given that the Boogie program verifier's source code is not available, in order to interface with the Boogie program verifier, JML4 must be able to translate JML-annotated java files into Boogie code, which will be fed into the Boogie program verifier.

7.3 Physical Constraints

The distributed program verification needs to be done by two remote machines:

7.3.1 The Dispatcher Server

The Dispatcher Server is a JML4Disco Server, which received messages from a client. These messages include VcProgram objects, which in turn are broken down buy the Dispatcher into Vcs. These Vcs are then sent to other servers - Prover Servers - to be proven in parallel.

7.3.2 The Prover Server

In addition to having JML4Disco installations, Prover Servers have various Theorem Provers such as Isabelle - sometimes customized for use with JML4. A Prover Server accepts a message from a Dispatcher Server, which includes a Vc object. It returns the results of using various provers to verify the Vcs.

12. Interfaces

12.1 User Interfaces

The end-user will be using the Eclipse SDE with JML4 Disco. Setup of the project can be done through the configuration panel of Eclipse as well as configuration files.

12.2 Software Interfaces

The system shall interface with the Eclipse JDT components, the JML4 Project (including its provers) as well as Boogie. JUnit will also be used for testing. Cashing is done by storing information to binary files at the moment, but a database remains a possibility in future iterations.

12.3 Communication Interfaces

The JML4 Disco project should be able to considerably improve the verification time of Extended Static Checking utilising either remote resources available over either LAN or WAN.

15. Applicable Standards

JML4 Disco is an evolution of JML4. Given that JML4 is an Eclipse-based Integrated development Verification Environment, built on top of the Eclipse Java Development Tooling, JML4 Disco should be roughly complaisant with the Eclipse Development Conventions and Guidelines.

15.1 Coding Standards

Since JML4 Disco is an evolution of the JML4, which in turn is being built on top of the Eclipse Java Development Tooling (JDT) JML4 Disco should follow Eclipse's Coding Conventions.

16. Internationalisation and Localisation

The JML4 Disco Project follows the Internationalisation conventions of the Eclipse PDE. The process of Internationalisation is done by NLS-enabling the product. Enabling a product for National Language Support (NLS) ensures the product is designed for national language function and uses proper APIs to handle national language data. At the moment, no significant translation is needed since most of the information returned to the user through the JML4 Disco Project does not need translation.
